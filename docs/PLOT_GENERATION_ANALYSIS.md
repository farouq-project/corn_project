# Plot Generation Analysis

**Date:** 2026-07-20  
**Status:** Audit only. No code changes.

---

## What Is a "Plot"?

In field research, a **plot** is a physical area of land planted with one genotype for one replication. In an RCBD design:

- A trial with **K genotypes** and **R replications** has **K × R plots per environment**.
- Each plot has a **physical plot number** (sequential, field-assigned) and carries a **genotype assignment** (which variety is planted there).
- In a multi-location trial (M environments), total plots = K × R × M.

---

## Plot Number: Where Does It Actually Come From?

There are **three different concepts of "plot number"** in this codebase, stored in different places:

### Concept A: `trial_plots.plot_number` (Gen 2 system — stranded)

**Table:** `trial_plots`  
**Column:** `plot_number` (int)  
**Source:** Generated by `RcbdService` via `TrialPlotController::generateRcbd()`.

This is the **physically meaningful** plot number — the sequential label a researcher would write on a stake in the field. It comes from the RCBD randomization algorithm.

```
trial_plots: {
  trial_id,
  environment_id,
  genotype_id,
  entry_number,       ← position in the entry list (stable across envs)
  plot_number,        ← physical field position (different per env due to randomization)
  randomization_order ← order within the RCBD randomization
}
```

**Critical:** `trial_plots` is **not connected** to `observation_records`. There is no FK from `observation_records` to `trial_plots`. These systems are entirely separate.

### Concept B: `observation_records.plot_no` (Gen 3 system — active)

**Table:** `observation_records`  
**Column:** `plot_no` (varchar 20)  
**Source:** User-typed in the "Tambah Baris" wizard. No validation against any master list.

This is a **free-text label** chosen by the researcher at data-entry time. It may or may not correspond to the physical plot number from the field layout. Examples of what users type: `"1"`, `"A1"`, `"B3"`, `"Plot-1"`.

**Unique constraint:** `(environment_id, season_id, plot_no, replication)` — ensures no duplicate entries for the same physical plot within one environment/season, but only if `plot_no` is typed consistently.

### Concept C: `entry_number` (Gen 3 grid — derived)

**Source:** `trial_genotypes.entry_number` — assigned when a researcher adds a genotype to a trial.

In the current `grid()` endpoint implementation (Batch 5), auto-created records use `plot_no = String(entry_number)`. This is a reasonable proxy but creates a hidden assumption: **the researcher's plot_no in the field corresponds to the trial's entry_number**. This is not guaranteed.

## How Plot Numbers Are Currently Created

### Path 1: "Tambah Baris" wizard

1. Researcher types any string in the "No Plot" field in Step 1 of the wizard.
2. The string is validated only as: `required | string | max:20`.
3. It is stored verbatim in `observation_records.plot_no`.
4. No linkage to any pre-existing plot layout.

### Path 2: Excel Import

1. Excel file has a "No Plot" column.
2. `PhenotypingImportService` reads the cell value verbatim.
3. Stored as `plot_no` in `observation_records`.
4. No validation against a master list.

### Path 3: Grid auto-create (Batch 5 — new)

1. User clicks a cell in an empty grid row.
2. `handleCellChange` in `data-pengamatan/page.tsx` calls `POST /v1/phenotyping/records` with `plot_no = row.plot_no`.
3. `row.plot_no` comes from the `grid()` endpoint: either the existing record's `plot_no` or `String(entry_number)` for empty rows.
4. Result: new record has `plot_no = "1"`, `"2"`, `"3"` etc. (matching entry_number).

## What Information Is Needed to Generate the Spreadsheet

For the spreadsheet to generate all rows without any pre-existing data:

| Required | Source | Currently Available? |
|---|---|---|
| List of genotypes for trial | `trial_genotypes` with `entry_number` | ✓ If researcher assigns them |
| List of environments for trial | `trial_environments` | ✓ If researcher links them |
| Replication count | `trials.replications` | ✓ Always set |
| Plot number per genotype | `trial_genotypes.entry_number` used as proxy | ⚠ Proxy only — not a true field plot number |
| Connection record → trial | `observation_records.trial_id` | ✗ **MISSING** |

### What Is Currently Missing

1. **`observation_records.trial_id`** — There is no way to know which trial an observation record belongs to. This means:
   - The grid heuristic (genotype+env+rep) can match records from the wrong trial.
   - Records auto-created by the grid have no trial context.
   - The aggregate endpoint cannot filter by trial.

2. **A single authoritative "plot number" per genotype per trial** — Currently `entry_number` is used as a proxy. This works if researchers accept that entry 1 = plot 1 = "No Plot 1". If the field uses RCBD randomization (different physical plot numbers per block/environment), the proxy breaks.

3. **Import-to-environment mapping** — The import pipeline maps "Environment" to `environment_conditions` (treatment types: Normal/Shading/Drought), not to `environments` (physical locations like "Sumedang", "Garut"). This makes imported records invisible in the grid.

## Replication Analysis

### Where Replication Is Stored

- `trials.replications` — the number of replications for the whole trial (e.g. 3)
- `observation_records.replication` — the replication number for this specific record (1-based integer)
- `trial_plots.trial_block_id` — in Gen 2, each block = one replication

### How Replication Is Created

Gen 3 (active system): Replication is typed by the researcher in "Tambah Baris" Step 1, or read from the "R" column in Excel import. It is validated as `required | integer | min:1`. No upper bound enforced against `trials.replications`.

The `grid()` endpoint generates rows for `rep = 1 .. trial.replications`, using `trials.replications` as the upper bound. This is the correct behavior.

### Relationship with Plots and Environments

In RCBD:
- One replication = one complete set of all genotypes in one block
- Same genotype appears in R different blocks (replications), in each environment
- So plot count per environment = K genotypes × R replications

The `grid()` generates: `K × M × R` total rows (K=genotypes, M=environments, R=replications). This is correct for a fully crossed design.

## Environment Analysis

### Can One Trial Have Multiple Environments?

**Yes.** `trial_environments` is a junction table allowing many-to-many.

### Can One Plot Exist in Multiple Environments?

**In Gen 2 (TrialPlot):** No — `trial_plots` has `environment_id` as a direct FK. Each plot belongs to exactly one environment.

**In Gen 3 (ObservationRecord):** An observation record has one `environment_id`. The "same genotype, replication" appears as separate records per environment. The grid generates separate rows for each (genotype × environment × replication) combination, which is correct.

### How Environment Connects to Trial → Observation

```
Trial
  └─ trial_environments (junction)
       └─ Environment (physical location: Sumedang, Garut, ...)
            └─ observation_records (WHERE environment_id = env.id)
                 └─ observation_values
```

**Import breaks this chain:** Imported records have `environment_id = NULL` and carry `environment_condition_id` (treatment type) instead. The treatment type is a separate concept from physical location.

## Generation 2 Plot System — Why It Was Not Adopted

The Gen 2 (`trial_plots`) system is architecturally more complete. It supports:
- True RCBD randomization
- Physical block/row/column positions
- Plot status (active/damaged/missing/excluded)
- Link between observation and specific plot

However, it was not adopted for the active Gen 3 system because:
1. It references `phenotype_variables` (Gen 1 trait list) not `characteristics` (Gen 3 trait list)
2. It has no frontend implementation
3. The data-entry workflow (via `PlotObservationController`) requires a plot to exist before observations can be entered — creating the same "need to pre-create rows" problem
4. No import pipeline exists for it

The key insight from Gen 2 that should be incorporated into Gen 3: **TrialPlot as the authoritative plot source**. If `observation_records` had a `trial_plot_id` FK (and the grid generated rows from `trial_plots`), the plot number, genotype assignment, and environment context would all come from one authoritative source.
