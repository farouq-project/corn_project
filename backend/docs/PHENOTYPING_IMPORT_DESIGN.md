# Phenotyping Bulk Import — Design (Phase 2)

Status: **scaffolding only**. Migration, models, and skeleton services exist;
`upload`/template download work; `validate`/`preview`/`confirm`/`rollback`
return `501 Not Implemented`. This document describes the intended Phase 2
implementation so the TODOs can be filled in directly.

The pipeline mirrors `Import/InventoryImportService` (4-step:
upload → normalize/validate → preview → confirm/rollback), adapted for the
"Data Pengamatan" spreadsheet format and the `observation_records` /
`observation_values` EAV schema.

## Spreadsheet format

One row per plot/replication:

| No Plot | Kode Gen | Gen | Environment | R | TT (cm) | PD (hari) | ... |
|---------|----------|-----|-------------|---|---------|-----------|-----|

- Static columns: `No Plot`, `Kode Gen`, `Gen`, `Environment`, `R` (replication).
- One column per **active** `Characteristic`, ordered by `display_order`,
  headed `"{code}"` or `"{code} ({unit})"`. Generated by
  `PhenotypingImportService::buildTemplateHeaders()` /
  `PhenotypingObservationTemplateExport`.
- `Gen` (genotype name) is informational only — `Kode Gen` is the lookup key.

## Tables

- `phenotyping_import_batches` — one row per uploaded file. Status enum:
  `uploaded → parsing → parsed → validating → validated → confirmed →
  importing → completed | partial | failed | rolled_back`.
- `observation_import_staging` — one row per spreadsheet data row.
  - `raw_data` (json): `{header => cell value}` exactly as read from the sheet.
  - `normalized_data` (json, nullable): normalized fields after step 2:
    `{ plot_no, genotype_code, environment_code, replication, resolved:
    {genotype_id, environment_id, season_id}, values: {characteristic_code:
    float|null} }`.
  - `status`: `pending | valid | warning | invalid`.
  - `errors` / `warnings` (json): arrays of `{field, message}`.
  - `imported_observation_record_id`: set on confirm, used for rollback.

## Step 1 — Upload & parse (`uploadAndParse`)

Implemented: stores the file (`imports/phenotyping/Y/m/{uuid}.xlsx`,
sha256 hash), creates the batch row (`status=parsing` → `parsed`).

**TODO**: read the sheet with `Maatwebsite\Excel` (e.g. `ToCollection`
or `HeadingRowFormatter::default('none')` + first row as headings), and for
each data row create an `ObservationImportStaging` row with
`raw_data = [header => cell]`, `status = 'pending'`. Update
`batch.total_rows` and set `status = 'parsed'`.

Stop at the first fully-empty row (matches the template's trailing blank
example row).

## Step 2 — Normalize & validate (`normalizeAndValidate`)

For each `pending` staging row:

1. **Normalize** via `PhenotypingNormalizationService`:
   - `normalizePlotNo(raw['No Plot'])`
   - `normalizeGenotypeCode(raw['Kode Gen'])`
   - `normalizeReplication(raw['R'])`
   - For each active characteristic column, `normalizeNumericValue(raw[header], $decimalPlaces)`.
   - Environment code: trim/uppercase (no dedicated normalizer yet — reuse
     `NormalizationService`'s generic string handling or add one).

2. **Validate** via `PhenotypingValidationEngine::validateObservationRow()`,
   using the pre-loaded `genotypeCache` / `environmentCache` /
   `characteristicCache`:
   - **Errors** (→ `status = invalid`):
     - `plot_no`, `genotype_code`, `environment_code`, `replication` missing.
     - `genotype_code` not found in `genotypeCache`.
     - `environment_code` not found in `environmentCache`.
     - `replication` not a positive integer.
     - Duplicate `(environment_id, season_id, plot_no, replication)` — check
       both within the file (other staging rows in this batch) and against
       existing `observation_records`.
   - **Warnings** (→ `status = warning` if no errors):
     - Characteristic value outside a plausible range (e.g. negative where
       not expected) — define soft bounds per characteristic group, or skip
       if no bounds are configured.
     - All characteristic values empty for a row (nothing to import).
   - Otherwise `status = valid`.

   `resolved.environment_id` / `resolved.genotype_id` are filled from the
   caches; `resolved.season_id` is taken from the resolved environment's
   `season_id` (matches `ObservationRecord::creating()` denormalization).

3. Persist `normalized_data`, `status`, `errors`, `warnings` per row.
   Update batch counters (`valid_rows`, `invalid_rows`, `warning_rows`,
   `total_rows`) and set `batch.status = 'validated'`.

## Step 3 — Preview (`getPreviewData`)

Return paginated staging rows (optionally filtered by `status`), plus a
summary `{total, valid, warning, invalid}` — same shape as
`InventoryImportController::preview()`. No DB writes.

## Step 4 — Confirm (`confirmImport`)

Inside a DB transaction, for each staging row with `status in (valid,
warning)` and not already imported:

1. `ObservationRecord::create([...])` using `normalized_data` (plot_no,
   genotype_id, environment_id, replication, recorded_by = confirming user).
   `season_id` is auto-derived by the model's `creating()` hook.
2. For each `characteristic_code => value` in `normalized_data.values` where
   `value !== null`, `ObservationValue::create([...])`.
3. Set `staging_row.imported_observation_record_id`.
4. `AuditService::logCreated($record, ...)` per record (or one
   `logAction('phenotyping_import_confirmed', ...)` summary entry — match
   whichever convention `InventoryImportService::confirmImport` already uses).

Update `batch.imported_rows`, `batch.confirmed_by/confirmed_at`,
`batch.status = 'completed'` (or `'partial'` if some rows were skipped).

If a row's `(environment_id, season_id, plot_no, replication)` already
exists as an `ObservationRecord` (re-import), `update` it and its values
instead of creating a duplicate — mirrors the inline-edit `updateOrCreate`
behavior in `ObservationRecordController::update`.

## Rollback (`rollback`)

Soft-delete every `ObservationRecord` referenced by
`staging_rows.imported_observation_record_id` for this batch (and their
`ObservationValue` rows, or rely on cascade if soft-deletes don't cascade).
Set `batch.is_rolled_back = true`, `rolled_back_at`, `rolled_back_by`,
`status = 'rolled_back'`. Log via `AuditService::logAction()`.
